# 세부 정규분포 전환 시간 계측 설계

## 작업 목적
- 세부 정규분포 전환 시간을 실제 브라우저 기준으로 계측한다.
- 개발 중 전환 시간과 캐시 hit/miss 여부를 즉시 확인할 수 있게 한다.
- 계측 결과를 바탕으로 추가 튜닝 포인트를 빠르게 판단할 수 있게 한다.

## 현재 상태
- 세부 정규분포 차트는 응답 캐시와 계산 캐시를 사용한다.
- 하지만 실제 전환 시간이 어느 구간에서 얼마나 걸리는지 화면에서 바로 알 수 없다.
- 현재 코드에는 `performance.now()` 기반 계측 로직이 없다.

## 목표
- 정규분포 토글 시작부터 시리즈 준비 완료까지의 시간을 측정한다.
- 응답 캐시 hit/miss 여부와 계산 캐시 hit/miss 여부를 함께 기록한다.
- 개발 모드에서만 마지막 측정값을 화면 배지와 콘솔 로그로 확인할 수 있게 한다.

## 설계

### 1. 측정 대상
- 시작 시점:
  - 사용자가 세부 차트에서 `정규 분포` 토글을 선택한 순간
- 종료 시점:
  - 정규분포용 `curveData`, `pointData`, `series`가 준비된 시점

### 2. 수집 항목
- 총 전환 시간(ms)
- 응답 캐시 hit/miss
- 계산 캐시 hit/miss
- 현재 `charId`, `scale`, `baseDate`

### 3. 표시 방식
- `import.meta.env.DEV`일 때만 우측 상단에 작은 성능 배지를 표시한다.
- 예시:
  - `정규분포 842ms · resp hit · compute miss`
- 개발 모드에서는 콘솔에도 같은 요약을 남긴다.
- 프로덕션에서는 배지와 로그를 숨긴다.

### 4. 구현 원칙
- 측정 상태는 `Chart.jsx` 내부에서만 관리한다.
- 성능 계측 자체가 다시 큰 렌더 비용이 되지 않도록 문자열 요약과 숫자만 저장한다.
- 정규분포 보기 이외의 모드에서는 측정 상태를 유지하되 화면 표시만 제한한다.

## 추가 튜닝 포인트
- 첫 uncached 진입이 여전히 느리면 KDE 계산을 더 쪼개거나 점 표시 밀도를 줄이는 옵션을 검토한다.
- cache hit인데도 느리면 ApexCharts 재렌더 비용 쪽을 더 줄여야 한다.

## 주요 변경 사항
- 정규분포 전환 시간 계측 추가
- 개발 모드 전용 성능 배지 추가
- 개발 모드 콘솔 로그 추가
